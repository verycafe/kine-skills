# Kine Pose Dev V3 Skill 蓝图

日期：2026-06-18

## 一句话定义

Kine Pose Dev V3 应该做成一个 **“动画导演 + 动作拆解 + AI 生成 + 严格质检”** 的 Phase A 工具。

它不是把原图扭来扭去。
它不是 Source Master / Rig 工作流。
它不是先拆部件再绑骨骼。

它要做的是：

```text
基于一张图片
 -> 先判断目标动作或状态变化怎样才合理
 -> 再生成真正像这个动作/状态变化的整张主体序列帧
 -> 最后只把通过严格视觉评估的帧放进 selected frames / GIF / strip / board
```

这里的“主体”不只等于角色。主体可以是人物、产品、道具、机械、车辆、UI 图标、环境元素、特效，或任何用户希望从静态图生成动作/变化序列的图像对象。

当主体是角色时，输出是 whole-character frame。
当主体不是角色时，输出是 whole-subject frame。

无论主体是什么，都不能把局部扭曲、组件图、mask、控制图、分层图、局部修补图伪装成最终动画帧。

## 完整步骤

| 步骤 | 做什么 | 借鉴谁的优点 |
| --- | --- | --- |
| 1. 读取图片和动作需求 | 明确输入图片里的主体是什么，用户要什么动作、状态变化、循环或一次性动作 | `$kine-pose` 的输入/任务边界 |
| 2. 先写动作合同 | 用文字拆解动作或变化：谁动、谁稳、力从哪里来、节奏是什么、哪些阶段必须出现 | `$kine-pose` 最核心优点：先判断动作是否成立 |
| 3. 选择动作门槛 | 按动作类型选择 gate；例如左右交替、接触支撑、道具交互、机械转动、表情变化、二级动态、序列连续性 | `$kine-pose` 的 action gates |
| 4. 做动画导演 Brief | 像动画导演一样说明节奏、重心、受力、镜头、循环方式、风格边界、不能出现什么错误 | V3 应该新增的导演层 |
| 5. 做语义部件理解 | 理解主体由哪些语义部分组成，哪些部分应该一起动，哪些部分必须保持稳定 | Layer.ai / Scenario 的优点：不是盲目改图，而是理解图像部件 |
| 6. 做每帧动作计划 | 每一帧写清楚移动主体、稳定主体、接触点、遮挡、重叠、连续性和硬拒绝项 | V3 的核心增强：把“合理动作”落实到每帧 |
| 7. 做软控制图 | 画粗略姿势、接触点、运动轨迹、旋转中心或状态标记，只作为 AI 生成参考 | `$kine-pose` 的 soft control 思路 |
| 8. 生成整张主体动作帧 | 用模型生成完整主体帧；角色生成完整角色，物体/机械/UI 生成完整主体状态帧 | `$kine-pose` 的 whole-frame 原则 |
| 9. 严格视觉评估 | 第一问永远是：这是否一眼像用户要的动作/变化？不像就拒绝 | `$kine-pose` 的最大优点，V3 必须继承 |
| 10. V3 语义质检 | 检查断裂、关节洞、漂浮、接触错误、遮挡错误、部件错位、源图姿势/状态回弹 | V3 + Layer/Scenario 的增强 |
| 11. 只选通过的帧 | 只有动作成立、主体身份稳定、结构合理、证据完整的帧才能进 `frames_selected` | `$kine-pose` 的 registry / selected-frame 规则 |
| 12. 生成 GIF / strip / board | 只用 `frames_selected` 合成预览，不能用候选图、控制图、修补图或 rejected 图 | `$kine-pose` 的证据链 |
| 13. 用户不认可就作废 | 如果用户说“不像目标动作/变化”，状态必须变成 `visual_rejected`，并保留 rejected 证据 | `$kine-pose` 的人工视觉优先规则 |

## 能力组合

```text
Kine Pose Dev V3
= `$kine-pose` 的动作判断骨架
+ Layer.ai / Scenario 的语义部件理解
+ V3 的每帧遮挡、接触、重叠、连续性质检
```

## 关键边界

- V3 只做 Phase A。
- V3 输出 selected frames、GIF、strip、board 和 QA evidence。
- V3 不做 Source Master、Component Ledger、Rig、Bones、Skin、IK、Timeline、Runtime Animation。
- V3 可以借鉴 Layer.ai / Scenario 的“部件理解”思路，但只能用于动作计划、生成提示、局部修补目标和质检。
- V3 不能把部件拆分、RGBA layer、mask、局部修补或本地扭曲结果当作最终帧。

## 最重要的失败教训

如果一个结果只是把原图扭曲、拉伸、挤压、局部变形，看起来有动静但不符合目标动作，它必须被判定为：

```text
visual_rejected
```

脚本通过不能覆盖人眼动作失败。
记录完整不能覆盖动作失败。
身份保持不能覆盖动作失败。

Phase A 的第一性原则永远是：

```text
先让动作或状态变化成立，再检查身份、风格、构图和证据链。
```
